IBM Tivoli Netcool/OMNIbus バージョン 8.1

MIB データのインポート

SNMP MIB データを MIB マネージャーへインポートする方法です。

始める前に

「設定」ウィンドウに、ベンダーの MIB で通常使用される、 MIB のロケーションを指定します。MIB が ベンダー固有のディレクトリーに存在するようにするとともに、関連するあらゆる装置番号が サブディレクトリーに存在するようにします。また、これらのディレクトリーには MIB のテキスト・ファイルのみ含まれるようにします。ほとんどの MIB は、NCHOME/omnibus/var/mibmanager/mibs/base ディレクトリーにインストールされ、すべての RFC MIB は、NCHOME/omnibus/var/mibmanager/mibs/rfc ディレクトリーにインストールされます。

インポート する MIB は、その他の MIB に従属していることがあります。インポート実行中に MIB マネージャーが、従属 MIB を検索できるようにするには、 検索パスを、従属 MIB ファイルのロケーションに 設定します。

手順

MIB データをインポートするには、以下の手順を実行します。

  1. 「インポート」をクリックするか、メニューから、 「ファイル」 > 「インポート」をクリックします。
  2. 「ディレクトリー」フィールドに、インポートする MIB モジュールのロケーションを指定します。
  3. 使用可能なサブディレクトリーで検出されたすべての MIB を MIB マネージャーが 解析するようにするには、「サブディレクトリーを全探索」チェック・ボックスを 選択します。
  4. 「インポート」をクリックします。

    「インポート 状況」ウィンドウは、MIB モジュールの解析時に表示されます。 このウィンドウの左側には、インポートされた MIB ファイルの解析中に さまざまな異なる MIB オブジェクト・タイプが検出されるたびに、それらの数が表示されます。

    MIB の解析が終わると、MIB マネージャーは、それぞれの MIB モジュールを レビューしてから、MIB マネージャーで検出されたすべての MIB オブジェクトを解析します。 すべてのオブジェクトが解析されると、MIB マネージャーは、それぞれの MIB をレビューしてから インポート・ステートメントを処理して、参照される MIB モジュールがないか探します。 参照される MIB モジュールが、既にインポートされた MIB モジュールに見つからない場合、MIB マネージャーは、検索パスに指定されたディレクトリーを 検索します。「インポート状況」ウィンドウの右側には、 インポートされた MIB で検出されたインポート・ステートメントの状況を示す、 以下の表のようなアイコンが表示されます。
    アイコン 説明
    インポート・ステートメントが解決されなかったことを 示します。
    インポート・ステートメントが解決されたことを 示します。
    インポート・ステートメントで参照される MIB モジュールを 解決できなかったことを示します。

    インポート・ステートメント・ツリーを展開すると、インポート・ステートメントを使用して他の MIB モジュールを参照している MIB モジュールを表示できます。

    インポートされたすべての MIB ファイルが処理されると、「インポート が完了しました」が、状況フレームの上部に 表示されます。インポート中に生成された状況メッセージのすべてを表示するには、 「状況履歴の 表示」をクリックし、「ビュー」ドロップダウン・リストを使用して フィルター・オプションを選択します。

  5. 「閉じる」をクリックして、「インポート 状況」ウィンドウを閉じます。

タスクの結果

「MIB モジュール」ビューは、新たにインポートされた MIB モジュール名を 表示するために、自動的に最新表示されます。「OID ツリー」ビューは、 新たにインポートされた、すべての MIB オブジェクトを表示するために、自動的に最新表示されます。 MIB マネージャーは、ツリーを再作成するために、 ツリーの親ノード (iso、ccitt、および joint-iso-ccitt) に対して、 繰り返し検索を実行して、子ノードがないか確認します。これは、関連付けられた子を持たない子ノードが検出されるまで 行われます。エラーが検出されなければ、すべて項目を取り込み済みの MIB ツリーが「OID ツリー」ビューに表示されます。これには、 解析されたオブジェクトが入っています。

「OID ツリー」ビューでオブジェクトをクリックすると、オブジェクトが「MIB モジュール」ビューに自動的に表示されます。「詳細」ビューには、 選択したオブジェクトに関する詳細情報が表示されます。

インポートされた MIB モジュールは、xml ファイルに保管され、MIB モジュールの構造を 表します。xml ファイルは NCHOME/omnibus/var/mibmanager/data ディレクトリーにあります。

未解決の インポート・ステートメントが検出された場合、欠落した MIB の 担当ベンダーに連絡して、欠落したファイルを検索パスに追加してから インポートを繰り返します。

注:

ベンダーは、重複するオブジェクト名を 指定したり、RFC で指定されたオブジェクト名と 同じオブジェクト名を指定することがあります。

例えば、 system という名のオブジェクトは、OID が 1.3.6.1.2.1.1 の モジュール SNMPv2-MI' に定義され、別の system という名のオブジェクトは、 OID が 1.3.6.1.4.1.311.1.1.3.1.1.23 のモジュール WINDOWS-NT-PERFORMANCE に定義されています。 また、sysDescr という名のオブジェクトは、system.1 として モジュール SNMPv2-MIB に定義され、別の sysFileReadOperationsPerSec という名のオブジェクトは、 system.1 としてモジュール WINDOWS-NT-PERFORMANCE に定義されています。 この場合、子オブジェクトが関連付けられる先の親オブジェクトは 不明確です。

親の名が重複するオブジェクトが見つかった場合、 MIB マネージャーは、同じ MIB モジュール内に定義された親を探そうと します。前述の例で、これは WINDOWS-NT-PERFORMANCE です。どちらの親も同じ MIB モジュールに定義 されていない場合、MIB マネージャーは検索を行い、オブジェクトが依存する他の MIB モジュールに親が定義されていないかを確認します。 また、MIB マネージャーはその親を参照する別の MIB モジュールを識別し、 最も一般的に参照される親の名前を選択します。ただし、この結果、 MIB ツリーにデータが誤って取り込まれ、OID が誤って算出される可能性があります。このため、MIB マネージャーは、重複する オブジェクト名を検出する度に、警告をログに記録するとともに重複を解決するための アクションをデバッグ・ファイルに記録します。

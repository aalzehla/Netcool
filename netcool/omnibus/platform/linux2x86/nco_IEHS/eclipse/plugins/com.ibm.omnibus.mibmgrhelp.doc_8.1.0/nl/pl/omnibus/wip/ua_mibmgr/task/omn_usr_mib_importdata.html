IBM Tivoli Netcool/OMNIbus wersja 8.1

Importowanie danych MIB

Sposób importowania danych MIB SNMP do programu MIB Manager.

Zanim rozpoczniesz

W oknie Preferencje należy określić położenie baz MIB, które są regularnie używanie przez dostawców baz MIB. Upewnij się, że bazy MIB znajdują się w katalogach konkretnych producentów, oraz że wszystkie powiązane numery urządzeń znajdują się w podkatalogach. Sprawdź również, czy te katalogi nie zawierają żadnych plików tekstowych niepowiązanych z bazami MIB. Większość baz MIB jest zainstalowana w katalogu NCHOME/omnibus/var/mibmanager/mibs/base, a wszystkie bazy MIB RFC są zainstalowane w katalogu NCHOME/omnibus/var/mibmanager/mibs/rfc.

Importowane bazy MIB mogą być zależne od innych baz MIB. Aby się upewnić, że program MIB Manager może wyszukiwać zależne bazy MIB podczas procesu importowania, należy ustawić ścieżkę wyszukiwania do katalogu, w którym znajdują się pliki zależnych baz MIB.

Procedura

Aby zaimportować dane MIB:

  1. Kliknij opcję Import lub z menu wybierz opcję Plik > Import.
  2. W polu Katalog określ położenie modułów MIB, które chcesz zaimportować.
  3. Aby program MIB Manager przeprowadził analizę wszystkich baz MIB znalezionych w dostępnych podkatalogach, zaznacz pole wyboru Przeglądaj podkatalogi.
  4. Kliknij przycisk Importuj.

    W trakcie analizy modułów MIB wyświetlane jest okno Status importowania. W lewej części tego okna jest wyświetlana liczba różnych typów obiektów MIB w trakcie ich wykrywania podczas analizy importowanych plików MIB.

    Po analizie baz MIB, program MIB Manager przegląda każdy moduł MIB i analizuje każdy obiekt MIB, który udaje mu się znaleźć. Po zakończeniu analizy wszystkich obiektów, program MIB Manager przegląda każdą bazę MIB i przetwarza instrukcje importowania, aby znaleźć moduły MIB, do których istnieją odwołania. Jeśli jakiekolwiek wyszczególnione moduły MIB nie mogą być znalezione w uprzednio zaimportowanych modułach MIB, program MIB Manager przeszukuje katalogi podane w ścieżce wyszukiwania. Po prawej stronie okna Status importowania wyświetlane są ikony pokazujące status instrukcji importowania, które zostały znalezione w zaimportowanych bazach MIB, jak to pokazano w poniższej tabeli:
    Ikona Opis
    Wskazuje, że instrukcja importowania nie została przetworzona.
    Wskazuje, że instrukcja importowania została przetworzona.
    Wskazuje, że moduł MIB, do którego istnieje odwołanie w instrukcji importowania, nie może być przetworzony.

    Można rozwinąć drzewo instrukcji importowania, aby wyświetlić moduły MIB, które używają instrukcji importowania do odwołania się do innych modułów MIB.

    Status Importowanie zakończone jest wyświetlany w górnej części ramki statusu, gdy wszystkie importowane pliki MIB zostały przetworzone. Można wyświetlić wszystkie komunikaty statusów wygenerowane w trakcie importowania, klikając opcję Wyświetl historię statusu i wybierając z listy rozwijanej Widok opcję filtrowania.

  5. Kliknij opcję Pomiń, aby zamknąć okno Status importowania.

Wyniki

Widok Moduły MIB jest odświeżany automatycznie, aby wyświetlić nowo zaimportowane nazwy modułów MIB. Widok Drzewo OID jest odświeżany automatycznie, aby wyświetlić wszystkie nowo zaimportowane obiekty MIB. Aby odbudować drzewo, program MIB Manager przeszukuje nadrzędne węzły drzewa (iso, ccitt i joint-iso-ccitt) dopóki nie znajdzie węzła potomnego, który nie ma powiązanych elementów potomnych. Jeśli nie zostaną wykryte żadne błędy, pełne drzewo MIB zostaje wyświetlone w widoku Drzewo OID, który zawiera przeanalizowane obiekty.

Kliknięcie obiektu w widoku Drzewo OID powoduje automatyczne wyświetlenie obiektu w widoku Moduły MIB. Widok Szczegóły wyświetla szczegółowe informacje na temat wybranego obiektu.

Zaimportowane moduły MIB są zapisywane w pliku XML, który reprezentuje strukturę danego modułu MIB. Pliki XML znajdują się w katalogu NCHOME/omnibus/var/mibmanager/data.

Jeśli zostaną wykryte nieprzetworzone instrukcje importowania, należy skontaktować się z dostawcą odpowiedzialnym za brakujące bazy MIB, dodać brakujące pliki do ścieżki wyszukiwania oraz powtórzyć proces importowania.

Uwaga:

Producenci mogą czasami podawać zduplikowane nazwy obiektów lub takie, które są identyczne z nazwami obiektów określonymi w RFC.

Na przykład, obiekt o nazwie system jest zdefiniowany w module SNMPv2-MIB z identyfikatorem OID 1.3.6.1.2.1.1, a inny obiekt o nazwie system jest zdefiniowany w module WINDOWS-NT-PERFORMANCE z identyfikatorem OID 1.3.6.1.4.1.311.1.1.3.1.1.23. Również obiekt o nazwie sysDescr jest zdefiniowany jako system.1 w module SNMPv2-MIB i innych obiekt o nazwie sysFileReadOperationsPerSec jest zdefiniowany jako system.1 w module WINDOWS-NT-PERFORMANCE. W takim przypadku jest niejasne, z którym obiektem nadrzędnym powiązane są obiekty potomne.

Gdy zostaje znaleziony obiekt ze zduplikowaną nazwą elementu nadrzędnego, program MIB Manager próbuje znaleźć element nadrzędny, który jest zdefiniowany w tym samym module MIB. W tym przykładzie jest nim WINDOWS-NT-PERFORMANCE. Jeśli żaden z obiektów nadrzędnych nie jest zdefiniowany w tym samym module MIB, program MIB Manager przeszukuje inne moduły MIB, aby sprawdzić, czy element nadrzędny jest zdefiniowany w modułach, od których zależny jest obiekt. Program MIB Manager identyfikuje również inne moduły MIB, które odwołują się do tego elementu nadrzędnego i wybiera najczęściej występującą nazwę elementu nadrzędnego. Może to jednak spowodować niepoprawne zapełnienie drzewa MIB i niepoprawne obliczenie identyfikatorów OID. Dlatego każda zduplikowana nazwa obiektu znaleziona przez program MIB Manager jest rejestrowana jako ostrzeżenie. Ponadto program ten zapisuje w pliku debugowania działania podjęte w celu rozstrzygnięcia tej duplikacji.

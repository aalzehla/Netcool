Importando dados MIB

Como importar dados MIB SNMP para o MIB Manager.

Antes de Iniciar

Na janela Preferências, especifique os locais dos MIBs usados regularmente pelos MIBs do fornecedor. Assegure-se de que os MIBs estejam localizados nos diretórios específicos para fornecedores e que qualquer número de equipamento associado esteja localizado em subdiretórios. Assegure-se também que os diretórios não contenham nenhum arquivo de texto não MIB. A maioria dos MIBs é instalada no diretório NCHOME/omnibus/var/mibmanager/mibs/base e todos os MIBs de RFC são instalados no diretório NCHOME/omnibus/var/mibmanager/mibs/rfc.

Os MIBs que você estiver importando poderão ser dependentes de outros MIBs. Para assegurar que o MIB Manager possa procurar MIBs dependentes enquanto estiver executando a importação, configure o caminho da procura para os locais dos arquivos MIB dependentes.

Procedimento

Para exportar dados MIB:

  1. Clique no Importar ou a partir do menu, clique em Arquivo > Importar.
  2. No campo Diretório, especifique o local dos módulos MIB que você deseja importar.
  3. Para fazer com que o MIB Manager analise todos os MIBs localizados nos subdiretórios disponíveis, marque a caixa de seleção Atravessar subdiretórios.
  4. Clique em Importar.

    A janela Status de Importação será exibida à medida em que os módulos são analisados. O lado esquerdo desta janela exibe um número diferente de tipos de objeto MIB à medida em que cada um é descoberto durante a análise dos arquivos MIB importados.

    Após a análise dos MIBs, o MIB Manager revisa cada módulo MIB e analisa todos os objetos MIB que forem localizados. Após a análise de todos os objetos, o MIB Manager revisa cada MIB e processará as instruções de importação para localizar os módulos MIB referenciados. Se algum módulo MIB referenciado não puder ser localizado nos módulos MIB importados anteriormente, o MIB Manager procurará nos diretórios especificados no caminho da procura. O lado direito da janela Status de Importação exibe ícones para mostrar o status das instruções de importação localizados nos MIBs importados, conforme mostrado na tabela a seguir:
    Ícone Descrição
    Indica que a instrução de importação não foi resolvida.
    Indica que a instrução de importação foi resolvida.
    Indica que o MIB Module referenciado na instrução de importação não pôde ser resolvido.

    É possível expandir a árvore de instrução de importação para mostrar quais módulos MIB estão usando as instruções de importação para referenciar outros módulos MIB.

    Importação Concluída será exibida na parte superior do quadro de status quando todos os arquivos MIB importados tiverem sido processados. É possível visualizar todas as mensagens de status geradas durante a importação clicando em Visualizar Histórico de Status, usando a lista suspensa Visualização para selecionar uma opção de filtro.

  5. Clique em Descartar para fechar a janela Status de Importação.

Resultados

A Visualização MIB Modules é atualizada automaticamente para mostrar os novos nomes dos módulos MIB importados. A visualização Árvore de OID é atualizada automaticamente para mostrar todos os objetos MIB recentemente importados. Para reconstruir a árvore, o MIB Manager procura nos nós-pai da árvore (iso, ccitt e joint-iso-ccitt) recursivamente por nós-filho até localizar um nó-filho que não tenha filhos associados. Se não forem detectados erros, uma árvore MIB totalmente preenchida será exibida na visualização OID Tree, que contém objetos analisados.

Clicar em um objeto na visualização OID Tree automaticamente exibe o objeto na visualização MIB Modules. A visualização Detalhes exibe informações detalhadas sobre o objeto selecionado.

Os módulos MIB importados são armazenados em um arquivo xml que representa a estrutura daquele módulo MIB. Os arquivos xml estão localizados no diretório NCHOME/omnibus/var/mibmanager/data.

Se instruções de importação não resolvidas forem detectadas, entre em contato com o fornecedor responsável pelo(s) MIB ou MIBs ausentes, inclua o arquivo ou arquivos ausentes ao caminho da procura e repita a importação.

Nota:

Os fornecedores às vezes especificam nomes do objeto em duplicidade ou especificam um nome do objeto idêntico a um nome do objeto especificado em um RFC.

Por exemplo, um objeto com o nome de system é definido no módulo SNMPv2-MI' com OID 1.3.6.1.2.1.1 e outro objeto com o nome de system é definido no módulo WINDOWS-NT-PERFORMANCE, com OID 1.3.6.1.4.1.311.1.1.3.1.1.23. Além disso, um objeto com o nomesysDescr é definido como system.1 no módulo SNMPv2-MIB e outro objeto com o nome de sysFileReadOperationsPerSec is definido como system.1 no módulo WINDOWS-NT-PERFORMANCE. Neste caso, fica incerto qual objeto-pai está associado com os objetos-filho.

Quando um objeto com um nome de pai duplicado é localizado, o MIB Manager tenta localizar o pai que está definido no mesmo módulo MIB. No exemplo anterior, trata-se do WINDOWS-NT-PERFORMANCE. Se nenhum dos pais estiver definido no mesmo módulo MIB, o MIB Manager procurará ver se o pai está definido em outros módulos MIB dos quais o objeto é dependente. O MIB Manager também identifica outros módulos MIB que referenciam o pai e seleciona o nome do pai mais comumente referenciado. Entretanto, isto poderá resultar em uma árvore MIB preenchida incorretamente e OIDs calculados incorretamente. Por esta razão, toda vez que o MIB Manager localiza um nome de objeto duplicado, ele registra um aviso e registra as ações tomadas para resolver a duplicação em um arquivo de depuração.

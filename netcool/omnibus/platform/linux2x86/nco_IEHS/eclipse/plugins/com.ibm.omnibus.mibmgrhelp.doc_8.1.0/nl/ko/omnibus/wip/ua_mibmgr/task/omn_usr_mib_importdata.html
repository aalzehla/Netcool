IBM Tivoli Netcool/OMNIbus 버전 8.1

MIB 데이터 가져오기

SNMP MIB 데이터를 MIB Manager로 가져오는 방법입니다.

시작하기 전에

환경 설정 창에서 공급업체 MIB가 정기적으로 사용하는 MIB의 위치를 지정하십시오. MIB가 공급업체 고유 디렉토리에 있는지 그리고 연관된 장비 번호가 서브디렉토리에 있는지 확인하십시오. 또한 이 디렉토리에 비MIB 텍스트 파일이 없는지 확인하십시오. 대부분의 MIB는 NCHOME/omnibus/var/mibmanager/mibs/base 디렉토리에 설치되고 모든 RFC MIB는 NCHOME/omnibus/var/mibmanager/mibs/rfc 디렉토리에 설치됩니다.

가져오는 MIB가 기타 MIB에 종속될 수 있습니다. 가져오기를 수행하는 동안 MIB Manager가 종속 MIB를 검색할 수 있도록 하려면 검색 경로를 종속 MIB 파일의 위치로 설정하십시오.

프로시저

MIB 데이터를 가져오려면 다음을 수행하십시오.

  1. 가져오기를 클릭하거나, 메뉴에서 파일 > 가져오기를 클릭하십시오.
  2. 디렉토리 필드에서 가져오려는 MIB 모듈의 위치를 지정하십시오.
  3. MIB Manager가 사용 가능한 서브디렉토리에 있는 모든 MIB를 구문 분석하도록 하려면 서브디렉토리 순회 선택란을 선택하십시오.
  4. 가져오기를 클릭하십시오.

    MIB 모듈이 구문 분석되면서 가져오기 상태 창이 표시됩니다. 이 창의 왼쪽에서는 다른 MIB 오브젝트 유형 수가 표시되고, 각각은 가져오는 MIB 파일을 구문 분석하는 동안 발견된 것입니다.

    MIB를 구문 분석한 후 MIB Manager가 각 MIB 모듈을 검토하며 모듈이 찾는 모든 MIB 오브젝트를 구문 분석합니다. 모든 오브젝트를 구문 분석한 후에는 MIB Manager가 각 MIB를 검토하고 가져오기 명령문을 처리하여 참조된 MIB 모듈을 찾습니다. 이전에 가져온 MIB 모듈에서 참조된 MIB 모듈을 찾을 수 없으면 MIB Manager가 검색 경로에 지정된 디렉토리를 검색합니다. 가져오기 상태 창의 오른쪽에서는 다음 테이블에 표시된 대로 가져오는 MIB에서 찾은 가져오기 명령문의 상태를 표시하는 아이콘을 표시합니다.
    아이콘 설명
    가져오기 명령문이 해석되지 않았음을 표시합니다.
    가져오기 명령문이 해석되었음을 표시합니다.
    가져오기 명령문에서 참조되는 MIB 모듈을 해석할 수 없음을 표시합니다.

    가져오기 명령문 트리를 확장시켜 기타 MIB 모듈을 참조하기 위해 가져오기 명령을 사용하는 MIB 모듈을 표시할 수 있습니다.

    가져오는 모든 MIB 파일의 처리가 완료되면 가져오기 완료가 상태 프레임 맨 위에 표시됩니다. 보기 상태 히스토리를 클릭하고 보기 드롭 다운 목록을 사용하여 필터 옵션을 선택하면 가져오기 도중 생성된 모든 상태 메시지를 볼 수 있습니다.

  5. 닫기를 클릭하여 가져오기 상태 창을 닫으십시오.

결과

MIB 모듈 보기는 자동으로 새로 가져온 MIB 모듈 이름을 표시하도록 새로 고쳐집니다. OID 트리 보기는 새로 가져온 MIB 오브젝트를 모두 표시하도록 자동으로 새로 고쳐집니다. 트리를 다시 빌드하기 위해 MIB Manager는 연관된 하위가 없는 하위 노드를 찾을 때까지 트리의 상위 노드(iso, ccitt, joint-iso-ccitt)에서 하위 노드를 반복적으로 검색합니다. 오류가 없으면, OID 트리 보기에 완전히 채워진 MIB 트리가 표시되고, 여기에는 구문 분석된 오브젝트가 포함되어 있습니다.

OID 트리 보기에 있는 오브젝트를 클릭하면 MIB 모듈 보기에서 오브젝트가 자동으로 표시됩니다. 세부사항 보기에서는 선택한 오브젝트에 대한 상세 정보를 표시합니다.

가져오는 MIB 모듈은 해당 MIB 모듈의 구조를 표시하는 XML 파일에 저장됩니다. XML 파일은 NCHOME/omnibus/var/mibmanager/data 디렉토리에 있습니다.

해석되지 않은 가져오기 명령문을 발견하면, 누락된 MIB를 담당하는 공급업체에 연락하고 검색 경로로 누락된 파일을 추가한 후 가져오기를 반복하십시오.

참고:

공급업체에서 가끔 중복된 오브젝트 이름을 지정하거나, RFC에서 지정한 오브젝트 이름과 같은 오브젝트 이름을 지정합니다.

예를 들어 OID가 1.3.6.1.2.1.1인 system 오브젝트가 SNMPv2-MI' 모듈에서 정의되고, OID가 1.3.6.1.4.1.311.1.1.3.1.1.23인 또 다른 system 오브젝트가 WINDOWS-NT-PERFORMANCE 모듈에서 정의됩니다. 또한 sysDescr 오브젝트가 SNMPv2-MIB 모듈에서는 system.1로 정의되고, 또 다른 sysFileReadOperationsPerSec 오브젝트가 WINDOWS-NT-PERFORMANCE에서 system.1로 정의됩니다. 이런 경우 하위 오브젝트와 연관된 상위 오브젝트가 분명하지 않습니다.

중복 상위 이름을 갖는 오브젝트가 발견되면 MIB Manager가 동일한 MIB 모듈에서 정의된 상위를 찾으려고 합니다. 이전 예제의 경우는 WINDOWS-NT-PERFORMANCE입니다. 상위가 모두 동일한 MIB 모듈에서 정의되지 않은 경우에는 MIB Manager가 검색을 통해 오브젝트가 종속된 다른 MIB 모듈에 상위가 정의되었는지 여부를 확인합니다. 또한 MIB Manager는 상위를 참조하는 다른 MIB 모듈을 식별하고 가장 많이 참조되는 상위 이름을 선택합니다. 그러나 이로 인해 MIB 트리가 잘못 채워지고 OID가 잘못 계산될 수 있습니다. 따라서 MIB Manager가 중복 오브젝트 이름을 찾을 때마다 경고를 로그하고 중복을 해결하기 위해 수행한 조치를 디버그 파일에 기록합니다.
